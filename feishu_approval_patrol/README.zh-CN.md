# 飞书审批巡检自动化用例

本目录用于介绍一个基于 Faray Lark Gateway 的飞书审批巡检用例。它适合把重复、低风险、规则明确的审批节点交给自动化处理，同时把金额不明、职责不清或需要人工判断的事项保留给审批人。

## 适用场景

团队在飞书中有多类审批流，例如合同审批、包材/样品审批、RU 相关样品审批等。审批人每天需要反复检查待办队列，判断哪些可以直接处理，哪些必须人工复核。

这个用例把巡检动作标准化：

1. 定时查询当前登录账号的飞书待审批队列。
2. 按标题、表单字段和审批节点过滤目标审批。
3. 对明确满足规则的审批执行动作。
4. 对不满足规则或归属不清的审批只记录 blocker，不越权处理。
5. 输出简短结果，包含处理数量、审批标题、流水号和阻塞原因。

## 核心原则

- 使用 Faray Lark Gateway，不走旧的 app-secret 直连路径。
- 只处理当前登录账号名下的待办任务。
- 先轻量查询队列，再按需读取审批详情。
- 默认保守：信息缺失、金额不明、节点归属不清时转人工。
- 不代替其他人的节点，不冒用其他审批人身份。
- 所有运行都要有简短摘要，便于回溯。

## 已覆盖的典型用例

### 1. 0 元合同自动审批

规则：

- 审批类型为合同申请或 Contract & Agreement Approval。
- 合同金额明确为 `0`。
- 合同类别不是框架类协议，或已明确属于可由部门负责人签署的普通合同。
- 当前待办节点属于当前登录账号。

自动动作：

- 对普通审批同意节点自动通过。
- 对提交/确认类节点也按同一规则处理，例如 `提交合同到审批流跟云盘`。

保留人工的情况：

- 金额不是 0。
- 金额字段缺失或无法解析。
- 明确为框架类协议。
- 当前节点分配给其他人。

### 2. 包材/样品审批巡检

规则：

- 只检查与 `包材`、`样板`、`样品` 相关的审批。
- 结合品牌规范、包材规范和审批附件做保守判断。
- 如果规范文件、图片证据或字段选择不完整，则只给出人工复核建议。

输出示例：

- 审批标题
- 流水号
- 涉及品牌/SKU/国家
- 建议动作：同意、驳回或人工复核
- 关键原因：例如文案疑似错误、条码缺失、认证标识未确认

### 3. RU 样品审批前置加签

规则：

- 审批必须同时命中 RU 信号和样品/物料信号。
- 只处理当前登录账号拥有的节点。
- 动作仅限新增前置审批人，不直接同意或驳回。

保守边界：

- US、EU 或其他非 RU 审批不处理。
- 标题或内容只有弱信号时不处理。
- 已有相同前置审批人时不重复加签。

## 推荐运行流程

```text
定时触发
  -> 查询当前账号待办
  -> 队列为空则记录最小状态并结束
  -> 过滤目标审批类型
  -> 读取实例详情
  -> 套用规则
  -> 只处理明确安全的节点
  -> 输出简短摘要
```

## 示例摘要

```text
飞书审批巡检完成
- fetched: 2
- matched: 1
- approved: 1
- manual: 0
- skipped: 1
- errors: 0
- approved: 合同申请 202606250039 / INFLUENCER COLLABORATION AGREEMENT - Nekteck x runalexia
- blocker: 后续提交节点分配给其他用户，当前账号不处理
```

## 接入前准备

1. 确认 Faray Lark Gateway 已登录。
2. 确认审批读写权限可用。
3. 明确当前自动化只作用于当前登录账号。
4. 把自动审批规则写成配置文件，而不是写死在临时脚本中。
5. 每次运行都写入结构化日志，至少包含 `fetched / matched / approved / rejected / manual / skipped / errors`。

## 安全边界

该用例不适合直接扩展为“所有审批自动通过”。每新增一个审批类型，都应该先定义：

- 哪些字段是必须确认的？
- 哪些情况可以自动处理？
- 哪些情况必须人工复核？
- 当前账号是否真的拥有该节点？
- 自动动作是否会影响合同、财务、法务或供应链责任？

如果任一问题无法回答，默认转人工。
